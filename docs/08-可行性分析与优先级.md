# 08 可行性分析与开发优先级

## 一、技术可行性

### 可行，但有约束

| 技术点 | 可行性 | 风险等级 | 说明 |
|--------|--------|---------|------|
| 微信小程序前端 | ✅ 完全可行 | 低 | WXML+WXSS+JS，无跨端框架，官方支持完善 |
| 微信云开发 | ✅ 可行 | 低 | 云函数+云数据库，免运维，与小程序天然集成 |
| 微信登录 | ✅ 可行 | 低 | wx.login + 云函数换openid，标准流程 |
| 选择题渲染 | ✅ 可行 | 低 | 标准组件，无技术难点 |
| 倒计时 | ✅ 可行 | 低 | setInterval + 云函数校准时间 |
| AI自动作答 | ✅ 可行 | 低 | 云函数内 setTimeout 模拟延迟 |
| 讨论实时消息 | ⚠️ 可行但有限制 | 中 | 云数据库 watch() 支持实时监听，但并发连接数有上限（50个/集合） |
| 多轮投票状态管理 | ⚠️ 可行但复杂 | 中 | 状态机较复杂，需仔细设计 |
| 随机事件系统 | ✅ 可行 | 低 | 纯逻辑判断，无技术难点 |
| 人类能力系统 | ✅ 可行 | 低 | 各能力独立，逐步实现即可 |

### 关键技术约束

**1. 云函数超时（20秒）**
- AI作答需模拟2-8秒延迟 → 需用云函数内 setTimeout，确保在20秒内完成
- 复杂推理逻辑（AI投票计算）需在20秒内完成 → 模板引擎足够快，LLM可能超时
- **应对**：MVP用本地模板引擎，无超时风险

**2. 实时消息上限**
- 云数据库 watch() 每个集合最多50个并发监听
- 单人游戏只有1个真实连接，无压力
- **应对**：MVP单人模式完全够用

**3. 题库内容**
- 需要80道高质量公考题（8类×10道）
- 每道题需：题目、4个选项、正确答案、解析
- **应对**：可从公开题库整理，或用AI辅助生成后人工校对

## 二、游戏设计可行性

### 核心玩法验证

**优势：**
- "反向狼人杀"概念新颖，市场有空白
- 公考题型有天然的"对错"判定，证据链清晰
- 单人+AI模式无需匹配，随时可玩

**风险：**

| 风险 | 严重度 | 说明 | 应对 |
|------|--------|------|------|
| 节奏重复 | 高 | 5轮都是"答题→揭→讨论→投票"，可能疲劳 | 随机事件+人类能力提供变化 |
| 人类压力过大 | 中 | 每轮都要答题+伪装+投票，焦虑感强 | 弃权权+静默模式提供喘息 |
| AI可预测 | 中 | 模板AI玩几局后模式被摸透 | 人格差异系数+随机扰动+LLM升级 |
| 题目难度波动 | 中 | 公考题难度差异大，影响公平性 | 按难度分级，控制出题顺序 |
| 终局无高潮 | 中 | 剩2人直接结束，缺紧张感 | 最终审判机制解决 |
| 被针对感 | 低 | 人类可能觉得"AI都针对我" | AI内讧机制解决 |

### 平衡性预判

**人类胜率估算（无能力系统）：**
- 简单模式（3AI）：~40%（AI少，但正确率低，容易被反向怀疑）
- 普通模式（4AI）：~25%（AI多，投票压力大）
- 困难模式（5AI）：~10%（几乎不可能）

**人类胜率估算（有能力系统）：**
- 简单模式：~55%（能力充裕）
- 普通模式：~35%（能力刚好够用）
- 困难模式：~20%（能力紧张，需要高超技巧）

**结论**：能力系统对平衡至关重要，不应推迟。

## 四、优先级排序与开发策略

### 优先级分层

```
P0 — 必须有，没有不能玩
├── Phase 1: 脚手架
├── Phase 2: 登录
├── Phase 3: 题库+游戏创建
├── Phase 4: 答题+揭示
└── Phase 6: 投票+淘汰

P1 — 应该有，影响核心体验
├── Phase 5: 讨论+AI发言
├── Phase 7: 循环+结算
└── Phase 8: 人类能力系统

P2 — 锦上添花，可后做
├── Phase 9: 随机事件
└── Phase 10: 打磨优化
```

### 推荐开发顺序（调整后）

原顺序的问题：Phase 8（人类能力）在太后面，能力系统是平衡核心，应与核心循环同步验证。

**调整建议：**

```
第一批（核心骨架）：Phase 1 → 2 → 3 → 4
  目标：能出题、能答题、能看到答案
  验证：单轮流程跑通

第二批（核心循环）：Phase 6 → 5 → 7
  目标：能投票、能淘汰、能多轮循环、能结算
  验证：完整一局可玩
  注意：先做投票(6)再做讨论(5)，因为投票是淘汰的核心机制

第三批（平衡与深度）：Phase 8 → 9
  目标：人类能力+随机事件
  验证：游戏有深度、有变化

第四批（上线准备）：Phase 10
  目标：体验优化、边界处理
  验证：可上线
```

### 为什么先做投票再做讨论？

投票（Phase 6）是淘汰的唯一机制，没有投票就没有游戏结果。讨论（Phase 5）是投票的辅助信息来源，但即使没有讨论，玩家也能投票（盲投）。所以：
- 先做投票 → 能跑通完整流程（虽然体验粗糙）
- 再做讨论 → 投票有依据，体验提升
- 这样每个阶段都能独立测试

## 五、关键风险与应对

| 风险 | 概率 | 影响 | 应对方案 |
|------|------|------|---------|
| 题库质量差 | 中 | 高 | Phase 3完成后立即小范围测试题库，收集反馈 |
| AI投票太准/太不准 | 高 | 中 | Phase 6完成后反复调参，人格权重是关键 |
| 游戏节奏无聊 | 中 | 高 | Phase 7完成后请3-5人试玩，根据反馈加事件/能力 |
| 微信审核不通过 | 低 | 高 | 避免敏感内容，公考题需去版权痕迹 |
| 云函数冷启动慢 | 中 | 低 | 预置并发实例，或前端预加载 |
| 单人开发进度失控 | 中 | 中 | 严格按Phase交付，每Phase有明确验收标准 |

## 六、最小可测试版本（MVP-MVP）

如果想用最短时间验证"这个游戏好不好玩"，可以只做：

```
Phase 1 + 2 + 3 + 4 + 6（精简版）
```

即：能登录 → 能创建游戏 → 能答题 → 能看答案 → 能投票淘汰

跳过讨论（Phase 5）和多轮循环（Phase 7），用单轮投票验证核心体验。

**预计工时：8-12天**
**验证目标：答题+投票这个核心循环是否有趣**

如果验证通过，再补讨论、多轮、能力、事件。
