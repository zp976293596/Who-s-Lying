# 07 开发步骤

## 总体原则
- 每个Phase控制在可独立运行、可测试的粒度
- 完成一个Phase后验证通过再进入下一个
- 优先跑通核心流程，再丰富细节

---

## Phase 1：项目脚手架
**目标：** 小程序能跑起来，能进入空白页面

- [ ] 用微信开发者工具创建小程序项目
- [ ] 配置 app.json 页面路由
- [ ] 创建 pages/ 目录结构（index、game、result、profile）
- [ ] 配置云开发环境
- [ ] 测试云函数调用（hello world）

**验收：** 模拟器能正常运行，4个页面可跳转

---

## Phase 2：首页与用户登录
**目标：** 用户打开小程序能看到首页，完成微信登录

- [ ] 实现微信登录流程（wx.login → 云函数获取openid）
- [ ] 创建 users 集合，首次登录自动建档
- [ ] 首页UI：标题、开始按钮、用户头像
- [ ] 赛博暗黑主题样式（全局wxss变量）

**验收：** 打开小程序自动登录，首页显示用户信息

---

## Phase 3：游戏创建与题库
**目标：** 能创建一局游戏，配置AI玩家，加载题目

- [ ] 创建 questions 集合，导入初始题库（每种题型10道，含正确答案和解析）
- [ ] 创建 ai_providers 集合，配置MVP默认AI（本地模板引擎，5种人格）
- [ ] 实现 createGame 云函数
  - 随机分配AI人格（ALPHA/BETA/GAMMA/DELTA/OMEGA）
  - 创建 games 记录，初始化所有玩家 humanPossibility=20
  - 创建第一轮 rounds 记录，随机选题
- [ ] 首页点击"开始游戏" → 创建游戏 → 跳转game页面

**验收：** 点击开始后，数据库中有游戏记录，包含AI配置和首轮题目

---

## Phase 4：答题与答案揭示
**目标：** 玩家和AI作答，公布正确答案，展示作答分析

- [ ] 游戏页面渲染题目（选择题，ABCD选项）
- [ ] 倒计时组件
- [ ] 玩家点击选项提交答案
- [ ] submitAnswer 云函数，记录 answers 集合（含作答时间、修改次数）
- [ ] AI玩家自动作答（根据人格模板的正确率决定是否答对，随机延迟2-8秒）
- [ ] 倒计时结束或所有人答完 → 进入揭示阶段
- [ ] 揭示页面：显示正确答案 + 每个人的答案 + 对错标记 + 作答时间
- [ ] 系统自动标记异常行为（答错、超时、犹豫）

**验收：** 能看到题目，提交答案，AI自动作答，揭示页显示所有人作答情况

---

## Phase 5：讨论与证据分析
**目标：** 玩家根据答题表现自由讨论，AI自动发表推理

- [ ] 讨论页面：上方显示本轮作答数据汇总（谁对谁错、谁快谁慢）
- [ ] 讨论消息列表（实时更新）
- [ ] 玩家发送文字消息
- [ ] AI自动生成讨论发言（基于答题数据推理，如"Player3第2题答错了，那题很简单"）
- [ ] AI内讧：AI之间也会互相质疑（OMEGA怀疑所有人，ALPHA关注数据，BETA从众），不只是针对人类
- [ ] 讨论倒计时

**验收：** 能看到作答数据，能发消息，AI会基于证据发言

---

## Phase 6：多轮投票与阈值淘汰
**目标：** 投票系统，累积人类可能性，超阈值淘汰

- [ ] 投票界面：显示所有存活玩家卡片 + 当前人类可能性指数
- [ ] 暗票机制：默认不公开谁投了谁，只显示每人获得几票
- [ ] 每人投出一个"最像人类"的目标 + 理由（AI自动生成理由）
- [ ] submitVote 云函数
- [ ] 投票结果处理：
  - 统计每人得票数
  - 更新被投票者 humanPossibility（+票数×权重）
  - AI根据本轮证据更新判断
- [ ] 嫌疑公示栏：游戏页面右侧常驻显示所有存活玩家 humanPossibility 排行，≥60分标红
- [ ] 多轮投票：一轮投票后若无人超阈值，可进行下一轮讨论+投票
- [ ] 阈值判定：humanPossibility ≥ eliminateThreshold → 该玩家被淘汰
- [ ] 淘汰揭示：被淘汰者身份公开（人类/AI）
- [ ] 人类被淘汰 → 游戏结束（AI胜）；AI被淘汰 → 检查剩余人数

**验收：** 能投票，暗票生效，嫌疑公示栏实时更新，超阈值淘汰正确

---

## Phase 7：多轮循环、最终审判与结算
**目标：** 完整的多轮游戏循环 + 最终审判 + 结算页面

- [ ] nextRound 云函数（创建新一轮，排除已淘汰玩家）
- [ ] 游戏结束判断：
  - 人类被淘汰 → AI胜
  - 存活到最后2人 → 进入最终审判
- [ ] 最终审判流程：
  - 存活2人时触发，专属UI（屏幕变红，矩阵震动效果）
  - 出一道题，两人同时作答
  - 答案揭示后进入对峙讨论（60秒）
  - 最终投票（明票）
  - 判定人类是否超阈值 → 人类胜利/AI胜利
- [ ] 结算页面UI：
  - 胜负结果
  - 人类身份揭示
  - 每轮投票数据回放
  - 人类可能性变化曲线
  - 人类能力使用记录
- [ ] 更新用户战绩（totalGames、wins等）
- [ ] "再来一局"功能

**验收：** 能完整玩一局，最终审判流程正确，结算页显示正确结果和数据回放

---

## Phase 8：人类优势系统
**目标：** 实现6种人类能力（含弃权），平衡1人对多AI的信息劣势

- [ ] games表players数组增加abilities字段
- [ ] 数据窥探：答题阶段可查看正确答案（按钮触发，弹窗显示）
- [ ] 信号屏蔽：投票时选择"屏蔽"，该票不计入投票得分
- [ ] 意识干扰：选择目标玩家，本轮humanPossibility额外+10
- [ ] 记忆覆写：答案揭示前可修改已提交答案
- [ ] 静默模式：讨论阶段发言不被AI分析
- [ ] 弃权：跳过一题不答，不产生异常得分（其他玩家可见"未作答"）
- [ ] 能力UI：游戏页面底部能力栏，显示剩余次数，灰色=已用完
- [ ] 能力使用记录：结算页展示人类使用了哪些能力
- [ ] 难度关联：简单模式能力次数+1，困难模式部分能力-1

**验收：** 人类可正常使用6种能力，能力效果正确，次数限制生效

---

## Phase 9：随机事件系统
**目标：** 实现5种随机事件，打破固定节奏，增加每局差异性

- [ ] 事件生成逻辑：每局开始时随机决定1-2轮触发事件（最后一轮不触发）
- [ ] 数据风暴：答题阶段触发，答案匿名展示（只显示"X人选A"，不显示谁选了什么）
- [ ] 矩阵扫描：揭示阶段触发，随机公开一名玩家近3轮答题记录
- [ ] 意识共振：讨论阶段触发，限时缩为30秒，每人只能发1条消息
- [ ] 协议覆盖：投票阶段触发，强制明票（显示谁投了谁）
- [ ] 时间裂隙：答题阶段触发，限时缩为15秒
- [ ] 事件公告UI：事件触发时弹出全屏公告，赛博暗黑风格特效
- [ ] rounds表增加event字段

**验收：** 每局随机触发事件，事件效果正确执行，不影响核心流程稳定性

---

## Phase 10：打磨与优化
**目标：** 体验优化，为上线做准备

- [ ] 加载状态与错误处理
- [ ] 动效完善（霓虹闪烁、投票动画、身份揭示、能力释放特效等）
- [ ] 音效（可选）
- [ ] 题库扩充
- [ ] 边界case处理（断线、超时、全部弃权等）
- [ ] AI推理质量调优（权重调整）
- [ ] 真机测试

**验收：** 真机体验流畅，AI推理合理，无明显bug
